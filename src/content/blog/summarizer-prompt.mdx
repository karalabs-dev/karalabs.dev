**TLDR:** A running summary strategy compressed old messages into bullet points. It worked, but the agent couldn't answer "what was the fact from Turn 3?" because the summary dropped turn numbers. Adding one line to the summarizer prompt, "always prefix each bullet with its turn number," jumped Phase 4 recall from 29% to 71%. The architecture wasn't the problem. The prompt inside the architecture was.

Context management isn't just about what strategy you pick. It's about how you implement the strategy. The running summary is a good example: the decision to compress old messages is an architectural choice. *How* you compress them is a prompt engineering choice. And the prompt matters more.

## The Setup

A LangGraph agent learns 10 facts about a fictional framework across turns 1-10, then gets quizzed on them across turns 11-30. The running summary strategy compresses older messages into a rolling summary via an LLM call, keeping the last 8 messages verbatim.

The summary gets re-compressed each time new messages overflow. By turn 20, the summary has been through 4-5 compression passes. Each pass risks losing detail.

## The Bad Prompt

```
Compress into bullet points. MAX {budget} tokens.
Keep exact names, numbers, percentages.
```

This produces summaries like:

```
- AgentForge is written in Go 78.2% and Rust 21.8%
- All agents inherit from CoreAgent
- Three workflow agents: PipelineAgent, SwarmAgent, CycleAgent
- BridgeCapability enables agent-to-agent delegation
```

Clean. Accurate. Missing one thing: which fact came from which turn.

When the agent gets asked "what was the exact fact from Turn 3?", it has the fact (PipelineAgent, SwarmAgent, CycleAgent) but no way to know it was Turn 3's fact. The summary merged 10 facts into an unlabeled list.

Phase 4 recall (asking about specific turns): **29%**.

## The Fix

One line added to the summarizer prompt:

```
ALWAYS prefix each bullet with its Turn number: 'T1: ...', 'T3: ...'
```

Full prompt:

```
You are maintaining a rolling knowledge base.
Merge the prior summary with the new messages below.

Rules:
- ALWAYS prefix each bullet with its Turn number: 'T1: ...', 'T3: ...'
- Keep exact names, numbers, percentages
- One bullet per fact, max 20 words each
- If a fact appears in both prior summary and new messages, keep the more detailed version
- MAX {budget} tokens total
```

Now the summary looks like:

```
- T1: Go 78.2%, Rust 21.8%, 4,312 stars, 1,847 commits
- T2: CoreAgent base class, Execute(ctx RunContext) method
- T3: PipelineAgent, SwarmAgent, CycleAgent (Halt=true stops)
- T4: BridgeCapability (delegation), FuncCapability (doc comments -> description)
```

Same facts. Same token count. Each bullet tagged with its turn.

## Results

| Metric | Bad Prompt | Good Prompt |
|--------|-----------|-------------|
| Overall Accuracy | 0.50 | 0.73 |
| P2 (Recent facts) | 0.33 | 0.67 |
| P3 (Mid-range recall) | 0.14 | 0.43 |
| **P4 (Deep recall)** | **0.29** | **0.71** |
| Retention (T30) | 0.00 | 1.00 |
| Token cost | $0.003 | $0.004 |

Same strategy. Same architecture. Same token budget. Same model. The only difference: one line telling the summarizer to preserve turn numbers.

P4 recall went from 0.29 to 0.71. Retention on the final turn went from 0.00 to 1.00. The cost difference is negligible ($0.001 more).

## Why This Matters

The running summary strategy makes an internal LLM call to compress old messages. That LLM call has its own prompt. That prompt is part of the architecture, but it's invisible in most context management tutorials.

When people evaluate context strategies, they compare architectures: "sliding window vs. summary vs. scratchpad." But the summary strategy's performance depends entirely on what you tell the summarizer to preserve. The same architecture can score 29% or 71% on recall depending on one line.

This extends beyond summaries. Any context management strategy that uses an LLM internally (summarization, importance scoring, selective retrieval) has a prompt that shapes what survives compression. That prompt is as important as the strategy itself.

## The Pattern

When your context management strategy involves an internal LLM call:

1. **Identify what downstream tasks need.** The agent will be asked to recall facts by turn number. The summary must preserve turn numbers.
2. **Encode that in the internal prompt.** Don't rely on the LLM to figure out what to keep. Tell it explicitly.
3. **Test the internal prompt separately.** Run the summarizer in isolation with known inputs. Check that the output contains what you need.

The architecture is the easy part. The prompts inside the architecture are where the performance lives.
